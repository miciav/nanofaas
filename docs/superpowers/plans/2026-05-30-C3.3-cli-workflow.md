# C3.3 — `cli` scenario come Workflow di Task (Implementation Plan)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Riscrivere lo scenario `cli` (`scenarios/cli_vm.py`) come `workflow_tasks.Workflow` di Task onesti, preservando comandi/ordine/task_id. È l'ultimo scenario; sblocca il ritiro del path recipe (`_execute_steps`) in C4.

**Architecture:** A differenza di k3s/helm/cli-stack, `cli` NON usa una recipe: i suoi step vengono da `scenario_planner.vm_backed_steps(request, include_bootstrap=...)` (= `vm_bootstrap_steps` + `vm_scenario_steps`), che produce `ScenarioPlanStep` **tutti host**, senza action/target/teardown. Conversione: `EnsureVmRunning` (per `vm.ensure_running`) eseguito per primo in `run()` → risolvi host → converti i restanti `ScenarioPlanStep` in Host `CommandTask` (placeholder `<multipass-ip>` risolti con l'host reale) → `Workflow(tasks=..., cleanup_tasks=[]).run()`. Nessun vm-task, nessuno speciale, nessun teardown.

**Tech Stack:** Python 3.11+, workflow_tasks, pytest, uv.

**Comandi:** controlplane `uv run --project tools/controlplane pytest <path>` (NIENTE `--no-cov`).

**Baseline pre-esistenti:** 3 (helm-stack autoscaling-env, run_all-bootstraps [entrambi socket multipass], tui_proxmox_cleanup).

**Template:** `scenarios/k3s_junit_curl.py` (pattern ensure-first/resolve-host), `scenarios/_workflow_assembly.py` (`build_setup`, `resolve_host_operation`, `CallableTask`), gli oracoli esistenti. Sorgente da sostituire: `scenarios/cli_vm.py`. Meccanismo step: `scenario/scenario_planner.py` (`vm_backed_steps`, `vm_bootstrap_steps`, `vm_scenario_steps`, `_step`/`_step_from_result`).

**Fatti verificati:**
- `cli` step (ordine): vm.ensure_running, vm.provision_base, repo.sync_to_vm, k3s.install, registry.ensure_container, k3s.configure_registry, cli.vm_e2e_flow. TUTTI `ScenarioPlanStep(summary, command, env, step_id)` host (nessun `action`, nessun target). Nessun `vm.down`/teardown.
- `vm.ensure_running` qui è un comando host (output dry-run di `vm.ensure_running`), NON un'action. In Workflow lo rappresentiamo con `EnsureVmRunning` (onesto + coerente con k3s; eseguito per primo in `run()`), e l'oracolo confronta il **task_id** (i comandi di EnsureVmRunning non si confrontano, come k3s).
- `cli.vm_e2e_flow` = host command `controlplane-tool cli-test run vm` con env da `_with_manifest_env(_vm_env(request))`.
- I comandi bootstrap host contengono placeholder `<multipass-ip:NAME>` risolti a runtime (`CommandResolver`) — vanno risolti con l'host reale dopo `EnsureVmRunning`, come in k3s.
- `build_cli_vm_plan(runner, request, include_bootstrap=True)` — il parametro `include_bootstrap` va preservato (E2eRunner lo usa nel chaining, linea ~617: `include_bootstrap=not vm_bootstrap_planned`). Quando `include_bootstrap=False`, gli step bootstrap (incl. ensure_running) sono OMESSI → in quel caso NON eseguire `EnsureVmRunning` e il Workflow contiene solo gli step scenario.

- [ ] **Step 1: Aggiungi `host_command_task_from_step`**

Aggiungi al modulo condiviso una funzione che converte un `ScenarioPlanStep` (host) in un `CommandTask` host, risolvendo i placeholder:
```python
def host_command_task_from_step(
    step,                      # ScenarioPlanStep (summary, command, env, step_id)
    *,
    resolver: CommandResolver,
    request: E2eRequest,
    vm,                        # VmOrchestrator
    ip_cache: dict[str, str],
    host_executor: HostCommandTaskExecutor,
) -> CommandTask:
    argv = resolver._resolve_command(list(step.command), request.vm, ip_cache, vm)   # noqa: SLF001
    env = resolver._resolve_env(dict(step.env), request.vm, ip_cache, vm)            # noqa: SLF001
    spec = CommandTaskSpec(task_id=step.step_id, summary=step.summary, argv=tuple(argv), target="host", env=env)
    return CommandTask(task_id=step.step_id, title=step.summary, spec=spec, executor=host_executor)
```
