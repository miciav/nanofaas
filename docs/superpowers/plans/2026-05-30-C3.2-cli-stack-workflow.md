# C3.2 — `cli-stack` come Workflow di Task (Implementation Plan)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Riscrivere `cli-stack` come `workflow_tasks.Workflow` di Task onesti, preservando comandi/ordine/task_id. È il più intricato di C3: usa un **secondo context** (`CliComponentContext`) per i componenti `cli.*` e ha **uno** step speciale (`cleanup.verify_cli_platform_status_fails` = "il comando deve fallire").

**Architecture:** Stesso pattern di k3s/helm via il modulo condiviso `scenarios/_workflow_assembly.py`, con due aggiunte: (1) `build_command_tasks` esteso con un `context_selector` per-componente (cli.* → `cli_context`, altrimenti context neutro); (2) un `special_handler` per `cleanup.verify_cli_platform_status_fails` che produce un `CallableTask` "expect-failure". `cli.*`, l'assembly e la verify RESTANO in controlplane (decisione confermata: prodotto-accoppiati).

**Tech Stack:** Python 3.11+, workflow_tasks, pytest, uv.

**Comandi:** controlplane `uv run --project tools/controlplane pytest <path>` (NIENTE `--no-cov`).

**Baseline pre-esistenti:** 3 (`test_e2e_runner.py::test_helm_stack_execute_resolves_vm_host_for_autoscaling_env`, `::test_run_all_bootstraps_vm_once_and_reuses_it`, `test_tui_choices.py::test_tui_proxmox_vm_loadtest_keeps_cleanup_phases_enabled`).

**Template (leggere):** `scenarios/k3s_junit_curl.py` + `scenarios/helm_stack.py` (Workflow), `scenarios/_workflow_assembly.py` (helper condivisi), i loro oracoli, e `scenarios/cli_stack.py` (versione recipe attuale da sostituire).

**Fatti verificati:**
- Recipe `cli-stack` (ordine): vm.ensure_running, vm.provision_base, repo.sync_to_vm, registry.ensure_container, images.build_core, images.build_selected_functions, k3s.install, k3s.configure_registry, namespace.install, cli.build_install_dist, cli.platform_install, cli.platform_status, cli.fn_apply_selected, cli.fn_list_selected, cli.fn_invoke_selected, cli.fn_enqueue_selected, cli.fn_delete_selected, cleanup.uninstall_control_plane, namespace.uninstall, cleanup.verify_cli_platform_status_fails, vm.down.
- Context dispatch (da `e2e_runner.plan_recipe_steps` linee 304-306): `cli_context if component_id.startswith("cli.") else context`.
- `cli_context` (da linee 294-301): `CliComponentContext(repo_root=Path(remote_dir), release=context.release, namespace=context.namespace, local_registry=context.local_registry, resolved_scenario=context.resolved_scenario, control_plane_endpoint=None)` — per cli-stack `control_plane_endpoint=None` (è `!= None` solo per two-vm-loadtest).
- `cli.fn_apply_selected` → `functions.register` SOLO se `scenario in LOADTEST_SCENARIOS`; **cli-stack NON lo è** → resta un comando cli normale (vm). NESSUN functions.register.
- Speciale unico: `cleanup.verify_cli_platform_status_fails` → `_expect_remote_failure` (esegue il comando vm; OK se return_code != 0; ERRORE "platform status unexpectedly succeeded after cleanup" se return_code == 0).
- `vm.down` → DestroyVm condizionale a `request.cleanup_vm` (no-op CallableTask con task_id="vm.down" altrimenti, per parità — come k3s).
- Le operazioni `cli.*` hanno `execution_target == "vm"` (girano il binario `nanofaas-cli` nella VM). Verifica leggendo `components/cli.py`.

- [ ] **Step 4: Riscrivi `cli_stack.py` come Workflow**

Ricalca `k3s_junit_curl.py`, con:
1. `_build_setup()` (riusa `build_setup` dal modulo condiviso). `remote_dir = runner.vm.remote_project_dir(vm_request)`.
2. Costruisci `cli_context = CliComponentContext(repo_root=Path(remote_dir), release=setup.context.release, namespace=setup.context.namespace, local_registry=setup.context.local_registry, resolved_scenario=setup.context.resolved_scenario, control_plane_endpoint=None)`.
3. `context_selector = lambda comp: cli_context if comp.component_id.startswith("cli.") else setup.context`.
4. `special_handler(operation)`:
   - se `operation.operation_id == "cleanup.verify_cli_platform_status_fails"`: ritorna un `CallableTask(task_id=operation.operation_id, title="Verify cli-stack status fails", action=<expect-failure>)` dove `<expect-failure>` esegue `runner.vm.exec_argv(vm_request, operation.argv, env=dict(operation.env) or None, cwd=remote_dir)` e solleva `RuntimeError("platform status unexpectedly succeeded after cleanup")` se `return_code == 0` (altrimenti ritorna). Riproduce `_expect_remote_failure`.
   - se `operation.operation_id == "vm.down"`: gestiscilo NEL cleanup (vedi punto 6), quindi qui ritorna `HANDLED` per saltarlo nei task principali. (Oppure escludi `vm.down`/`vm.ensure_running` dal loop come fa k3s.)
   - altrimenti `None` (routing di default).
5. `tasks = build_command_tasks(runner, request, setup, recipe, special_handler=special_handler, context_selector=context_selector)` — assicurati che `vm.ensure_running` e `vm.down` siano saltati dal routing di default (come in k3s: il loop salta questi op_id, oppure il special_handler ritorna HANDLED).
6. `EnsureVmRunning` eseguito per primo in `run()`; `cleanup_tasks = [DestroyVm(...)] if request.cleanup_vm else [CallableTask(task_id="vm.down", title="Skip VM teardown (--no-cleanup-vm)", action=lambda: None)]`.
7. `workflow_task_ids = ["vm.ensure_running"] + [t.task_id for t in workflow.tasks + workflow.cleanup_tasks]`.
8. Mantieni firma `build_cli_stack_plan(runner, request)` + `.run(event_listener=None)` (con `del event_listener`); mantieni `self.steps`/`task_ids` se load-bearing (grep tui/app.py, e2e_runner, test_scenario_builders).
9. Aggiungi cli_stack.py agli ignore ruff SLF001 in `pyproject.toml` se serve.

NB: assicurati che l'ordine dei task nel Workflow rispetti l'ordine recipe (il loop di `build_command_tasks` segue `compose_recipe`, che segue l'ordine dei component_ids). `cleanup.verify_cli_platform_status_fails` è nei task principali (NON cleanup), prima di `vm.down`.
