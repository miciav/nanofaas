# C3.1 — `helm-stack` come Workflow di Task (Implementation Plan)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Riscrivere lo scenario `helm-stack` come `workflow_tasks.Workflow` di Task onesti (stesso pattern del pilota `k3s-junit-curl`), preservando esattamente comandi/ordine/task_id. È il più semplice dei 3 scenari recipe rimasti: nessuno step speciale, nessun teardown VM.

**Architecture:** Identico a `scenarios/k3s_junit_curl.py` (già convertito), ma con la recipe `helm-stack` e SENZA: la verify k3s-curl, e SENZA `vm.down`/`DestroyVm` (la recipe helm-stack non ha `vm.down` — la VM resta attiva). Tutti gli step (incluso `loadtest.install_k6`, `loadtest.run`, `experiments.autoscaling`) sono `CommandTask` host/vm instradati per `execution_target`; `vm.ensure_running` → `EnsureVmRunning`.

**Tech Stack:** Python 3.11+, workflow_tasks, pytest, uv.

**Comandi:** controlplane `uv run --project tools/controlplane pytest <path>` (NIENTE `--no-cov`).

**Baseline fallimenti pre-esistenti:** controlplane 3 (`test_e2e_runner.py::test_helm_stack_execute_resolves_vm_host_for_autoscaling_env`, `::test_run_all_bootstraps_vm_once_and_reuses_it`, `test_tui_choices.py::test_tui_proxmox_vm_loadtest_keeps_cleanup_phases_enabled`). NB: il primo baseline è proprio un test helm-stack (fallisce per socket multipass mancante, non per logica) — deve restare fallito-uguale (non un NUOVO fallimento diverso).

**Template (leggere e replicare):** `tools/controlplane/src/controlplane_tool/scenario/scenarios/k3s_junit_curl.py` (versione Workflow già mergiata) e il suo oracolo `tools/controlplane/tests/test_k3s_junit_curl_workflow.py`. Riusa: `_build_setup`, il loop recipe→CommandTask (routing per `execution_target`), `_resolve_host_operation` (placeholder), `CallableTask`, `EnsureVmRunning`, gli executor.

**Recipe `helm-stack` (da `scenario/components/recipes.py`, preservare ordine):**
`vm.ensure_running, vm.provision_base, repo.sync_to_vm, registry.ensure_container, images.build_core, images.build_selected_functions, k3s.install, k3s.configure_registry, namespace.install, helm.deploy_control_plane, helm.deploy_function_runtime, loadtest.install_k6, loadtest.run, experiments.autoscaling`.

**Mappa step → Task:**
- `vm.ensure_running` → `EnsureVmRunning` (run in `run()`, poi risolvi host).
- TUTTI gli altri (provision_base, repo.sync, registry, images, k3s, namespace, helm, `loadtest.install_k6`, `loadtest.run`, `experiments.autoscaling`) → `CommandTask` instradato per `execution_target` (host→`HostCommandTaskExecutor(runner.shell)` con placeholder risolti; vm→`VmCommandTaskExecutor(OrchestratorVmRunner(vm_orch, vm_request))`).
- **Nessun** `vm.down`/`DestroyVm`, **nessuna** verify speciale, **nessun** `functions.register`. (Verifica leggendo la recipe e `operation_to_plan_step`/`plan_recipe_steps` che per helm-stack NON c'è alcun callback speciale: `loadtest.run`/`experiments.autoscaling` sono operazioni normali.)
